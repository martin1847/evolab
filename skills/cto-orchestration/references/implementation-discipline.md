# 实现与验证纪律（按需加载）

仅在写实现 goal、诊断 ReOpen、设计验证或收工核证时读取。主干判据见 `../SKILL.md` §3；这里保存操作展开与常见失败模式。

## 行为变更

- 默认把实验、性能优化和未批准的新行为藏在 env flag 后，默认 OFF。
- ReOpen 已明确批准的修复可以默认 ON，但在 goal 写明理由与兼容边界。
- 不为普通 bugfix 机械加 flag；按项目既有兼容策略判断。

## Measure before more

修 bug 前依次证明：

1. bug 在当前基线与真实输入上可复现；
2. 观测值确实受怀疑机制约束，而非查错数据源、日志串、session 或部署版本；
3. 执行真实到达怀疑代码；读代码只产生候选，日志、trace 或失败探针才证明路径。

绑定约束未证时，先派观测与复测，不发投机修复。第一刀后重新测量；修复会改变下一轮的绑定约束。

高不确定方向准备进入昂贵设计/实现时，先完成最便宜的只读证伪，再在 goal 写一行实际 probe 与读数，派发显式加 `--require-preflight`。runtime 会在启动 agent 前拒绝缺失、重复、占位或未执行状态；它只证明证据声明存在，不替代对读数真实性的独立核证。普通 bugfix、机械改动和用于取得该读数的 research goal 不启用此门。

## 病类收敛

root cause 一旦确认为一类模式，立即枚举代码库中所有同模式点，逐一标成安全 / 有嫌疑 / 需实验，一轮处理。不要只修眼前实例，等后续评审逐个发现同类点。

## 否定性结论与 Goodhart

只有决策依赖“未观察到 X”时才触发 known-positive 校准：先证明同一检测链能看见一个已知存在的 X；否则结论保持 `UNKNOWN`。验证动作数量不是验证有效性，坏验证会把不知道伪装成知道。

## 验证边界

- 不 mock 正在验证的边界；stub 掉被测函数只证明 stub。
- mock、打桩、stream smoke 与结构测试只能证明其覆盖范围，不证明真实部署路径。
- 本地绕过真 LLM、真队列、真数据库或真身份时，明确列入“未验证”，并安排部署后补验。
- 交付固定写清：验证了什么 / 没验证什么 / 剩余风险。

## 编辑新鲜度

rebase、另一 agent 写入同一 worktree、或任何会改目标文件的命令之后，重新读取再编辑。不要用旧上下文覆盖新文件；护栏提示 stale edit 时先重读，不绕过。
