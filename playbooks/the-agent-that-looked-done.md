# Playbook：那个"看起来做完了"的 agent

> 一个真实工作流的端到端复盘（脱敏）。讲的不是某个 bug，是一类**结构性盲区**怎么被发现、
> 怎么被根治。配套产物：`skills/cto-orchestration/references/watcher.sh`。

## 场景

编排模式下，我把一个后端改动派给执行 agent。它跑在一个终端复用器（tmux）的会话里，
我在它旁边挂了个 **watcher**：后台脚本每隔几十秒抓一次该会话的屏幕，判断 agent 是不是
还在忙；一旦闲下来就通知我去验收。这样我能并行盯好几个 agent，不用干等。

## 异常

watcher 报：会话 **IDLE，完成**。我去看产出——**什么都没有**。代码没改，测试没跑，
agent 像是根本没动过。但 watcher 言之凿凿说它"完成"了。

## 侦查：watcher 凭什么判完成？

逻辑很朴素：轮询屏幕里的"忙碌标记"（agent 工作时会显示的一个状态符）；连续两次抓不到
这个标记，就认定"不忙了 = 完成了"。

问题就藏在这步。**它是负向判定**——用"标记的缺失"推断完成。我去翻这次的屏幕快照：
是空的。agent 进程其实**根本没起来**（启动时崩了）。屏幕上自然没有忙碌标记。于是
"没有标记 → 完成"。

## 根因：这条链路没有失败信号

再往下挖一层，发现的是一个比"watcher 写错了"更深的东西：

> 通过终端复用器交互驱动子 agent，**整条链路没有任何一步会因为"agent 没干活"而失败。**

- 向会话发送按键：只要**会话**还在就返回成功——它不管对面有没有进程在接收。
- 抓取会话屏幕：只要**会话**还在就返回当前屏幕内容。agent 死了 / 没起来 / 退回了 shell，
  抓到的是"成功 + 一块空屏 / 一个 shell 提示符"，**不是一个错误**。

所以子进程崩了，发送和抓屏全都照常"成功"，只是屏幕是空的。**死亡被表达成了"成功 + 空"**，
而朴素的 watcher 把"空"（无忙碌标记）读成了"完成"。这不是 watcher 的笔误，是**用一个
没有失败信号的通道驱动 agent** 的固有后果。

（这正是 `meta/verification-honesty.md` 里"成功返回空 ≠ 成功"那条，在工程上的现身。）

## 修复：从"没有标记就算完成"翻成"必须有存活证据才算完成"

负向判定不可救，换正向判定。关键洞见：agent 是会话里的前台 TUI 程序，它**活着**的时候，
会话的前台进程就是它自己；它一旦退出，前台才会**交回 shell**。于是有了一个正向存活信号：

- 查会话的前台进程名。是 agent 的 TUI 进程 → 活着。变回了 shell（zsh/bash）→ **死了，不是完成**。
- agent 临时 shell-out 去跑 git / 测试时，前台是那个子进程，不是 shell，所以不会被误判成死亡。

在这之上把终态**分类**，而不是只有"完成 / 没完成"两态：

| 终态 | 判据 | 编排者动作 |
| --- | --- | --- |
| `DONE` | 无忙碌标记 **且** 前台仍是 agent 进程 | 去验收 |
| `AGENT DEAD` | 前台退回 shell | 重启，别当完成派去评审 |
| `WAITING INPUT` | 无忙碌标记但屏幕底部是交互提示 | 去回输入，不是完成 |
| `SUSPECTED HANG` | 忙碌标记还在但屏幕长时间零变化 | 去看一眼 |

完整实现见 `watcher.sh`，5 个退出码，编排者按码分流。

## 教训

1. **别从"没有失败信号"推断成功。** 完成的判据是正向证据（存活进程 + 真实产出），
   不是负向的"没看到忙"。
2. **盲区往往在通道本身，不在逻辑里。** watcher 的 if-else 没写错，错的是它信任了一个
   天生不会报错的通道。换通道前，先问"这条链路失败时，失败信号从哪来"——如果答案是"没有"，
   你的所有"成功"都要打折。
3. **第一次接一个新 agent，先校准它的存活信号长什么样**（活着时前台进程名是什么），
   和校准忙碌标记同理。
