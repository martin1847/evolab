# 验证诚实性：区分读到、推测、赌

> AI 最毒的失败不是报错——报错你会停、会知道。最毒的是**自信地猜**：把推测用确定的语气
> 说出来，听起来无懈可击，其实是脑补。在多 agent 系统里，一个自信的错误会被下游当事实接住，
> 越传越远。

## 三种置信度，说话时分开

每一句结论，问自己它的来源是哪一种：

- **读到的**：我实际读了那段代码 / 跑了那条命令 / 看了那个输出。可以当事实陈述。
- **推测的**：基于先验和局部线索的合理推断。必须**标成推测**——"我猜""大概率""没验证"。
- **赌的**：线索不足，纯赌。要么别说，要么明说在赌。

铁律两条：**不知道就说不知道，先去读再答**，别用 confident-sounding guess 填空；
**已经确定的事别 hedge**，加一堆"可能也许"是另一种不诚实。

## "成功返回空" ≠ 成功

最隐蔽的一类陷阱：调用**成功了**，但返回的是空 / 截断 / 占位内容，而你把这片空白当成
"成功但没内容"，用先验脑补了本该在那里的东西。

经典案例是通过终端交互驱动子 agent：发送指令和抓屏只要会话还在就"成功"，子进程死了、
没起来、退回 shell，你抓到的是"成功 + 空屏"，而不是一个错误。**整条链路没有任何一步会
因为"agent 没干活"而失败报错。** 于是死亡被读成完成。

解法是同一个原则的延伸：**别从"没有失败信号"推断成功，要正向证据。** 完成的判据不是
"没看到忙碌标记"，而是"看到了存活进程 + 真实产出"。（这条的完整工程实现见
`playbooks/the-agent-that-looked-done.md`。）

## 交付用三段式

任何"做完了"的汇报，拆成三段，别糊成一句"搞定了"：

1. **验证了什么**——真正跑过 / 读过的。
2. **没验证什么**——本地打桩绕过的、没覆盖到的路径、靠推断的环节。
3. **剩余假设与风险**。

为什么第 2 段最重要：本地用 stub 替代真实依赖时，最容易让一个只在真实环境暴露的 bug
溜过验证、逃到生产（一个被 stub 掩盖的未 await 的异步调用，本地全绿，线上才炸）。
诚实标注"这段是打桩跑的、真实依赖没测"，下游才知道要在部署后补验。

诚实不是谦虚的修辞，是让协作链路**可信**的前提——别人能不能接住你的结论，取决于你有没有
如实标好它的置信度。
