# 按可逆性分配自主权

> AI 协作里最难调的不是能力，是**自主权的边界**。给少了，什么都来问你，慢得像没有 agent；
> 给多了，不该动的它动了，一次不可逆操作就够你收拾半天。多数人靠"重要性""复杂度"切，
> 切不准。更锋利的一根轴是**可逆性**。

## 一根轴

把任何动作按"做错了能不能轻松撤回"分三档：

- **可逆（直接做，不请示）**：读代码、搜索、跑只读命令、改本地文件、写测试、跑本地基线、
  起草文档。撤回成本 ≈ 0。请示只会拖慢，纯损耗。
- **不可逆 / 对外（先说意图 + 影响面，等放行）**：`push` / 强推、迁移、生产部署、删分支、
  连生产库、对外发消息、覆盖已有数据。这些有 blast radius，撤回贵甚至撤不回。
- **边界模糊**：按**不可逆**处理。拿不准时高估风险，比低估便宜。

## 为什么是可逆性，不是重要性

重要性是连续的、主观的、当下难判的——"这个改动重要吗"经常要做完才知道。可逆性是
**结构性的、当场可判的**：这条命令有没有 `--force`、会不会写远端、删的东西有没有备份。
你不需要理解任务多重要，只需要看一眼操作能不能撤回。判据越客观，越能稳定执行、越能放心
委派给 agent。

## 双向适用

这条规矩同时治两个对象：

1. **治 agent**：写进 `AGENTS.md` 的 Work Modes（DO / THINK FIRST / REQUIRE APPROVAL
   三档），delegated agent 就知道哪些能自主、哪些必须停下报告。它越界的概率直接降下来。
2. **治编排者自己**：你自己也按这条走——可逆的批量推进不打断用户，不可逆的（替用户 push、
   删他的目录）先说清楚再动。**对已批准的事别再请示，对不可逆的事别隐性自主**，两个方向的
   病都治。

## 落地

- `AGENTS.md` 里列三档清单（见 `templates/AGENTS.md`）。
- 审批是**一次性**的：批准这次 push ≠ 批准下次。上下文换了，重新确认。
- 配套一条："做不可逆操作前，先看证据是否真支持这个具体动作"——一个看起来像已知故障的
  信号，可能根因不同。restart / 删除 / 改配置之前，先验证因果，别凭 pattern-match 下手。

可逆性不是胆小，是把"请示成本"花在真正不可逆的地方。可逆的地方全速，不可逆的地方刹一脚——
这就是一个人能稳稳指挥一群 agent 的底层节奏。
